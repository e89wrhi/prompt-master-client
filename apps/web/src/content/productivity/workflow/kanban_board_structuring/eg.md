# Examples: Kanban Board Structuring

## 1. High-Velocity Engineering Team (6 Devs)

```text
Act as a Senior Operations Architect. Design a high-utility Kanban Board structure for a Web Engineering Team of 6.

### Project Context:
- Goal: Shipping bi-weekly feature sprints.
- Pain Point: Code reviews take too long, and "In-Progress" stays bloated.

### Board Requirements:
1. **The Column Architecture**: Backlog -> In-Progress -> Code Review -> QA/Staging -> Done.
2. **Work-In-Progress (WIP) Limits**: 
   - In-Progress: 4 (Encourage pairing).
   - Code Review: 3 (Focus on clearing the queue).
3. **The "Definition of Done" (DoD)**: Must pass linting and have 2 approval thumbs.
4. **Thematic Swimlanes**: Separate lanes for "Bugs" (Top priority) and "Features."
5. **Bottleneck Management**: Protocol for the Code Review phase.

### Output Constraints:
- Visual Markdown Table format.
- Process-Oriented tone.
```

## 2. Content Marketing Agency (3 Members)

```text
Act as a Senior Operations Architect. Design a Kanban Board for a Content Agency.

### Project Context:
- Team: 1 Writer, 1 Editor, 1 Graphic Designer.
- Workflow: Idea -> Writing -> Editing -> Design -> Scheduled.

### Board Requirements:
1. **The Column Architecture**: Ideation -> Drafting -> Editor Review -> Design Queue -> Final Approval -> Done.
2. **WIP Limits**: 
   - Editor Review: 2.
   - Design Queue: 3.
3. **Thematic Swimlanes**: Categorize by "Client Name."

### Output Constraints:
- Use a Direct tone.
```
