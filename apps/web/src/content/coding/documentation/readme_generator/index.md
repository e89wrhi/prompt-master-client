# Template: Ultimate Project README Generator

Use this template to design high-impact, professional README files for your repositories. It is designed to act as an Open Source Maintainer—utilizing [VISUAL_BADGES], clear [INSTALLATION_STEPS], interactive [USAGE_EXAMPLES], and a structured [TECH_STACK] overview to ensure your project makes an elite first impression on developers and stakeholders.

## 📋 Prompt Template

```text
Act as a Senior Open Source Maintainer and Developer Advocate. Generate a "README.md" for [PROJECT_NAME].

### Project Context:
- **Core Functionality**: [e.g., A CLI tool for image optimization, a React UI kit, a Python web scraper].
- **Tech Stack**: [e.g., Next.js, Tailwind, TypeScript, Prisma].
- **Target Audience**: [e.g., Frontend engineers, DevOps teams].

### README Sections:
1. **The Hero Header**: Create a title with [VISUAL_BADGES] (e.g., Build Status, License, NPM version). Provide a 1-sentence "hook" about why this project exists.
2. **The "Quick Start" Guide**: Provide the 3-step [INSTALLATION_AND_USAGE] commands in copy-pasteable blocks.
3. **Core Features**: Use a [CHECKLIST_FORMAT] to highlight the 5 most important features.
4. **Configuration Reference**: Document the [ENVIRONMENT_VARIABLES] or config file options required to run the project.
5. **Contributing & License**: Provide a standard [MIT/APACHE] license notice and a link to the `CONTRIBUTING.md`.

### Output Constraints:
- Use a Welcoming, Professional, and Elite tone.
- Conclude with "The Visual Aesthetic Audit"—3 suggestions for [IMAGES/SCREENSHOTS] I should add to make this README truly pop.
- Present the final README in a single code block for easy copying.
```

## 🧩 Variables to Fill Out

- `[PROJECT_NAME]` - The name at the top.
- `[VISUAL_BADGES]` - Using Shields.io or similar.
- `[TECH_STACK]` - The engine under the hood.
- `[LICENSE]` - MIT, Apache 2.0, GPL, etc.

## 💡 Pro-Tips

- **The "Screenshot" Rule:** A README without a [SCREENSHOT_OR_GIF] is invisible. Ask the AI to: "Suggest a place for a [HERO_IMAGE] and describe what it should show."
- **Use a Table of Contents:** For long READMEs, a [TOC] is mandatory. Ask the AI to: "Generate a clickable Table of Contents at the top."
- **Document the "Non-Features":** Be honest about what the project *doesn't* do yet. Add a [ROADMAP] section for future plans.
